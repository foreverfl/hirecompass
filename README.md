## 1) 이번 매경 해커톤에서 **B2B로 가는 건 가능**한데, 조건이 있어요

대회 과제가 “뉴스 데이터를 활용해 **국민 생활 편의/사회 문제 해결**” 쪽이라서, **순수 내부업무용 B2B**(예: 사내 리포트 자동화)로만 가면 “임팩트”가 약해 보일 수 있어요.

대신 이렇게 잡으면 **B2B가 오히려 강점**이 됩니다:

* **B2B2C**: 기업/기관이 고객에게 제공하는 형태
  예) 은행/증권사 → “오늘 뉴스 기반 개인 상황별 행동 체크리스트(과장 없이 근거 링크 포함)”
  예) 지자체/공공기관 → “지역 재난/지원금/생활정보 뉴스 → 주민 행동요령 안내”
* **현장형 B2B**: 콜센터/상담사가 쓰는 “근거 있는 설명 + 다음 행동” _~생성
  예) 상담원이 기사 원문을 붙이면 → 핵심 요점, FAQ, “고객이 지금 해야 할 3가지”를 뽑아줌(출처 포함)

핵심은 **“최종 수혜자가 국민”**으로 보이게 설계하는 겁니다.

## 취업 문제를 ‘뉴스 데이터’로 푸는 차별점 3개 (여기서 1~2개만 박아도 달라집니다)

### 1) 알림이 아니라 **“행동 플랜” 알림**

일반 채용 알림 = “공고 떴습니다”
당신 서비스 = “공고 + 지금 당장 할 행동 3개 + 준비물 + 위험요소” (근거 기사 포함)

예시 알림 카드:

* “이번 주 IT 채용시장 변화(요약 3줄)”
* “지금 지원하면 유리한 포지션 유형(예: SRE/플랫폼/보안)”
* “오늘 할 일 체크리스트 5분짜리 2개 (이력서 한 줄 수정 / 포트폴리오 링크 점검)”
* “근거 기사 2개 + 근거 문장 하이라이트”

이게 ‘뉴스→액션’에 딱 맞습니다. ([매일경제][1])

### 2) RAG를 “추천”이 아니라 **‘근거 있는 해석’**으로 쓰기

뉴스로 취업을 돕는 핵심은 “채용 공고 모음”이 아니라,

* 산업/직무의 **수요 변화 신호**(감원/투자/규제/AI 도입/클라우드 전환)
* 그 변화가 개인에게 의미하는 **스킬/포지션 전략**
  을 뽑아주는 겁니다.

즉 RAG는:

* Top 기사 3~5개를 찾아서
* “요약”이 아니라 **전략 문장**을 만들게 하는 용도

### 3) 윤리/안전(Claude + 매경 가치) 포인트를 기능으로 보이게

해커톤 소개글 자체가 “안전·윤리”를 강조하니까 ([매일경제][2])
UI에 이런 고정 섹션을 넣으면 가점이 큽니다:

* **불확실성 표시**: “이건 추정입니다 / 확인 필요”
* **근거 공개**: 기사ID/날짜/핵심 근거 문장
* **편향 완화**: 특정 대학/성별/나이 차별 문구 생성 금지 + 자동 필터(프롬프트 가드레일)
* **법/규정 유의**: 채용 관련 규정(차별 금지 등)은 “확인 필요”로 처리

## “취업 알림봇(PWA)”를 해커톤 MVP로 현실적으로 만들기 (하루 데모 기준)

### MVP 화면 3개만

1. **프로필(로그인 없이)**

* 지역, 경력(0~3/3~5/5+), 관심직무(백엔드/프론트/SRE 등), 희망 고용형태
* *개인정보 최소* (로컬 저장)

2. **오늘의 시장 브리핑**

* 뉴스 기반 “이번 주 채용시장 변화 3줄”
* “내 직무에 영향 큰 키워드 3개” (예: “클라우드 비용절감 → FinOps/SRE 수요”)

3. **액션 카드**

* “이번 주 해야 할 일” 체크리스트 + 링크(학습/공고/이력서 템플릿 등)
* **근거 문장 하이라이트** 버튼

### 알림(봇) 형태

* PWA 푸시 알림(시간 되면)
* 아니면 데모에서는 **“알림 생성 화면”**만 구현해도 충분합니다(심사장에서 푸시까지 완벽히 안 해도 됨)

## 1) 수십만 기사면 벡터는 몇 개?

기본 규칙:

* **문서 1개(기사 1개) = 벡터 1개** (가장 단순)
* RAG 품질을 올리려고 본문을 쪼개면:

  * **기사 1개 = 청크 3~10개 = 벡터 3~10개**

즉, **벡터 개수 = 청크 개수**입니다.

### 대충 감 잡는 숫자(예시)

기사 **20만 개** 기준으로:

* (A) 기사 단위 임베딩: **20만 벡터**
* (B) 기사당 평균 5청크: **100만 벡터**
* (C) 기사당 평균 10청크: **200만 벡터**

현실적으로 “뉴스 기사”는 길이가 제각각이라, 평균 청크 수는 **3~8** 사이로 많이 잡습니다.
→ 그래서 1년치가 20만 기사면 **대략 60만~160만 벡터** 정도로 생각하면 큰 틀에서 맞아요.

**기준을 이렇게 잡으면 됨**

* “벡터 100만” = “청크 100만” = “RAG에서 검색 대상 100만 조각”
* 여기서 검색은 TopK 20~50만 뽑으면 됨

## 2) pgvector의 장점과 한계

### 장점

* **Postgres 하나로 끝**: 본문/메타/필터/트랜잭션/백업/운영이 전부 한 곳
* **하이브리드 검색이 쉬움**: `tsvector`(키워드) + `pgvector`(의미) 결과를 한 쿼리/한 파이프라인으로 합치기 좋음
* 이미 PG를 쓰면 도입 비용 낮음
* 해커톤/프로토타입에 매우 강함(디버깅 쉬움)

### 한계(현업에서 pgvector가 밀리는 지점)

* 벡터가 **수천만~수억** 수준으로 가거나 QPS가 높아지면, “벡터 전용 엔진” 대비 효율/비용이 불리해질 수 있음
* 검색 성능/튜닝이 DB 전체 성능(다른 쿼리, I/O, vacuum 등)과 얽힘
* 벡터만 전문적으로 최적화된 시스템만큼 옵션/분산성이 강하진 않음

요약하면:

* **작~중간 규모 RAG**: pgvector가 현실 최강
* **대규모/고트래픽 벡터 검색 플랫폼**: Milvus류가 유리해질 수 있음

## 3) Postgres + tsvector(FTS) + pgvector(HNSW) 각각 역할

이 조합은 사실상 **“키워드 검색 + 의미 검색 + 빠른 근사 최근접 탐색”** 세트입니다.

### Postgres

* “모든 데이터의 원장(source of truth)”
* 기사 본문, 제목, 날짜, 카테고리, 회사/직무 태그, URL 같은 **정형 메타데이터**
* 필터링/정렬/조인(예: 최근 7일, 경제/IT, 특정 키워드 포함 등)

### tsvector (FTS: Full Text Search)

* **키워드 기반 검색** 담당
* 장점: “정책명/회사명/직무명” 같이 **정확한 단어**를 잘 잡음
* 예: “청년내일채움공제”, “SRE”, “네이버 채용” 같은 건 벡터보다 FTS가 더 정확할 때가 많음
* 결과를 `ts_rank`로 랭킹 가능

### pgvector

* **의미 기반(semantic) 검색** 담당
* 사용자가 “요즘 취업이 어렵다, 어떤 스킬이 뜨냐”처럼 모호하게 물어도
  기사 의미가 비슷한 것을 찾아줌
* 즉, “단어가 정확히 안 맞아도” 관련 문서를 가져옴

### HNSW (pgvector 인덱스 방식 중 하나)

* 벡터 검색을 **빠르게** 하기 위한 인덱스
* 완전 정확한 최근접(Brute force)이 아니라 **근사(ANN)** 로 빠르게 TopK를 찾아줌
* 해커톤/서비스에서 사실상 필수 (벡터 수가 늘면 brute force는 느려짐)
